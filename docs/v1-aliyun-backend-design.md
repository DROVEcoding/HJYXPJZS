# V1：阿里云真实后台设计

## 1. 为什么改路线

之前的 V1 设计偏向 Supabase / Supabase 兼容后端。这个路线适合快速验证“登录、数据库、权限”这些概念，但你刚刚明确了真实方向：

```text
后台直接做在阿里云上。
```

所以 V1 应该从“前端直接连接后端平台”改成更通用、更真实的 SaaS 架构：

```text
前端页面 -> 自己的后端 API -> 阿里云 RDS 数据库
```

这条路线更接近以后真正给国内环评老师使用的产品。

## 2. 推荐阿里云架构

V1 推荐使用：

```text
前端：GitHub Pages 暂时承载静态页面
后端：阿里云 SAE 部署 Node.js / Express API
数据库：阿里云 RDS PostgreSQL
```

后续正式上线国内客户时，可以再把前端也迁到：

```text
阿里云 OSS + CDN + 已备案域名
```

## 3. 为什么不是浏览器直接连数据库

真实项目里，浏览器不能直接拿数据库账号和密码。

原因很简单：

```text
浏览器里的代码，用户都能看到。
```

如果把数据库密码放进前端，就等于把仓库钥匙贴在门口。正确做法是：

```text
数据库密码只放在阿里云后端环境变量里。
前端只请求自己的后端 API。
后端检查用户身份后，再去读写数据库。
```

这也是从 demo 走向真实 SaaS 的重要分界线。

## 4. V1 做什么

V1 只做“项目云端保存”的最小闭环：

- 用户可以注册；
- 用户可以登录；
- 用户可以退出；
- 用户可以保存当前生猪养殖环评项目；
- 用户可以读取自己的项目列表；
- 用户可以打开已保存项目继续编辑；
- 不同用户之间项目隔离。

V1 保存的数据包括：

| 数据 | 含义 |
|---|---|
| `form_data` | 用户填写的项目信息 |
| `assessment` | 环评类别辅助判断结果 |
| `drafts` | 已生成的章节草稿 |

## 5. V1 暂不做什么

V1 暂不做：

- 组织 / 团队；
- 管理员后台；
- 多角色权限；
- 向量库；
- 智能体生成；
- Word / PDF 导出；
- 支付；
- 正式生产域名和备案。

这些功能以后会做，但不能和 V1 混在一起。V1 的学习目标是理解：

```text
前端、后端、数据库、登录态、API、环境变量是怎么协作的。
```

## 6. 后端 API 设计

V1 后端提供这些接口：

| 接口 | 方法 | 用途 |
|---|---|---|
| `/api/health` | `GET` | 检查后端是否在线 |
| `/api/auth/register` | `POST` | 用户注册 |
| `/api/auth/login` | `POST` | 用户登录，返回 token |
| `/api/auth/me` | `GET` | 根据 token 获取当前用户 |
| `/api/projects` | `GET` | 获取当前用户的项目列表 |
| `/api/projects` | `POST` | 新建项目 |
| `/api/projects/:id` | `GET` | 打开单个项目 |
| `/api/projects/:id` | `PUT` | 更新单个项目 |

V1 先不做删除项目。删除是危险操作，等保存和权限跑通后再加。

## 7. 数据库设计

V1 推荐使用 PostgreSQL。

原因：

- JSON 字段能力强，适合保存表单、判断结果、草稿；
- 后续可以接全文检索、向量扩展或独立向量库；
- 更适合复杂 SaaS 数据结构。

### `users` 表

| 字段 | 类型 | 含义 |
|---|---|---|
| `id` | `uuid` | 用户 ID |
| `email` | `text` | 登录邮箱 |
| `password_hash` | `text` | 加密后的密码 |
| `created_at` | `timestamptz` | 创建时间 |

密码不能明文保存。后端必须用 `bcrypt` 之类的算法保存密码哈希。

### `projects` 表

| 字段 | 类型 | 含义 |
|---|---|---|
| `id` | `uuid` | 项目 ID |
| `user_id` | `uuid` | 所属用户 |
| `project_name` | `text` | 项目名称 |
| `category_id` | `text` | 分类 ID，例如 `pig-farming` |
| `form_data` | `jsonb` | 用户填写的项目资料 |
| `assessment` | `jsonb` | 环评类别辅助判断 |
| `drafts` | `jsonb` | 章节草稿 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

## 8. 登录和权限设计

V1 使用：

```text
邮箱 + 密码 + JWT token
```

流程是：

1. 用户注册邮箱和密码；
2. 后端把密码哈希后写入数据库；
3. 用户登录；
4. 后端验证密码，生成 JWT；
5. 前端把 JWT 存在浏览器本地；
6. 前端请求项目接口时，在请求头带上 JWT；
7. 后端解析 JWT，知道当前用户是谁；
8. 后端只允许用户读写自己的项目。

真正的权限边界在后端，不在前端按钮。

## 10. 本地开发方式

V1 本地开发时会同时开两个服务：

```text
前端静态服务：http://127.0.0.1:8788/
后端 API 服务：http://127.0.0.1:3001/
```

前端请求：

```text
http://127.0.0.1:3001/api/...
```

以后部署到阿里云后，前端再改成请求真实后端地址。

## 11. 阿里云部署方式

V1 的真实部署目标：

```text
阿里云 SAE 跑后端 API
阿里云 RDS PostgreSQL 存数据
```

用户需要在阿里云控制台完成：

- 开通 RDS PostgreSQL；
- 创建数据库账号和数据库；
- 设置白名单或 VPC 网络；
- 开通 SAE 应用；
- 配置后端环境变量；
- 部署后端代码。

Codex 可以做：

- 写后端代码；
- 写 SQL；
- 写环境变量示例；
- 写部署说明；
- 辅助你理解控制台界面；
- 根据你提供的公开地址修改前端配置；
- 本地测试。

Codex 不应该擅自做：

- 替你开通会产生费用的云资源；
- 替你确认付费订单；
- 接收或提交数据库密码、云平台 AK/SK、JWT_SECRET 等真实密钥；
- 把客户真实资料或完整语料提交到公开 GitHub。

## 12. 测试计划

自动测试：

- 后端健康检查测试；
- 注册接口测试；
- 登录接口测试；
- 未登录不能访问项目接口；
- 用户只能访问自己的项目；
- 前端核心逻辑测试继续保留。

手动测试：

1. 本地启动后端；
2. 本地启动前端；
3. 注册新账号；
4. 登录；
5. 填写生猪养殖项目；
6. 生成章节草稿；
7. 保存项目；
8. 刷新页面；
9. 登录后能看到项目列表；
10. 打开项目后表单和草稿恢复。

## 13. 验收标准

V1 完成时应满足：

- 本地前端能连接本地后端；
- 用户能注册、登录、退出；
- 用户能保存项目；
- 用户能读取自己的项目列表；
- 用户能打开已保存项目；
- 后端不会把密码明文保存；
- 数据库密码、JWT_SECRET 不进入 GitHub；
- 自动测试通过；
- README 有中文说明和文件用途表；
- 阿里云部署步骤有文档。
